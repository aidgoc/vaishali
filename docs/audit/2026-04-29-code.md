## Server-side guards

- **`auth_guard.py`** — `before_request` hook. Converts Guest hits on non-`allow_guest` `/api/method/vaishali.*` endpoints into HTTP 401 `SessionExpired` (instead of Frappe's default 403 HTML), so old PWA clients with only-401 recovery handlers redirect to login cleanly.
- **`visit_guard.py`** — `before_save` on Daily Call Report. Blocks new DCR creation from desk; PWA must set `frappe.local.flags.from_pwa=True`. On existing DCRs, silently reverts edits to `check_in_gps` / `check_out_gps` from non-PWA paths.
- **`service_call_guard.py`** — `before_save` on Service Call. Same PWA-only insert policy; protects truth-fields (`customer`, `channel`, `outcome`, `call_datetime`, `employee`) from desk edits.

## What's not here (gaps worth flagging)

- **No Workflows** — every state machine (Lead → Opportunity → Quotation → SO → Invoice; Warranty Claim → CAPA; Service Call lifecycle) is enforced in Python `doc_events` + view filters. Adds flexibility but means there's no visual state diagram in Desk; new joiners can't trace transitions without reading code.
- **No Server Scripts and only 1 Client Script** — all logic is in the versioned app, not the DB. Good for review/audit and disaster recovery, but means non-developer admins cannot tweak behaviour without a deploy.
- **No custom Reports** — analytics live in View Engine + Number Cards + dashboards. Power users who want classic Report Builder/Query Reports get nothing.
- **No Web Forms authored by Vaishali** — public-facing data entry is entirely PWA + Desk; no anonymous submissions. The 9 web forms in DB are stock Frappe.
- **Only 10 Frappe-native Notifications, 4 of which are scheduled** — alerting is concentrated in `notifications.py` Telegram pipeline (43 handlers), which is great for deliverability but bypasses Frappe's Notification Settings UI.
- **No `custom_field` fixtures in repo despite `fixtures = ["custom_field"]`** — 697 Custom Fields exist in DB but the export hasn't been committed; rebuilding from a fresh bench would lose all of them. Worth an audit.
- **DocPerm sprawl:** 472 Custom DocPerm rows is a large permission surface, not all version-controlled. ERPNext production permission tuning happens via Custom DocPerm (not in code) — same DR risk as fixtures.
- **Memory drift:** `tools` count, `ui.js` component count, and SW version all have drift between project memory and live repo. Memory should be regenerated.
- **`SLI` for tools mismatch:** `tools/__init__.py` aggregator references `system_config` count of 17 in the docstring `CLAUDE.md`, but the category lists 17 items including some that overlap with manage/list — worth verifying tool dispatch consistency.

## Risks / smells
- **Single-instance, no failover.** One EC2 box is a SPOF for ERP, FastAPI sidecar, MariaDB, and Redis. No read replica, no warm standby, no multi-AZ.
- **Tiny RAM (1.9 GB) on a production ERP** — already 350 MB swapped. One mis-tuned query or PDF export can OOM-kill gunicorn.
- **Plaintext secrets in `site_config.json`**: `aws_access_key_id` + `aws_secret_access_key`, `apollo_api_key`, `db_password`, `encryption_key`. These are world-readable to `frappe` group and to anyone who can read backups (which include `site_config_backup.json`). Consider IAM instance profile for AWS, secrets manager for the rest.
- **Backups land on the same root volume** (`/dev/root`, 49 GB, 36% full). If the box dies you lose everything except the **weekly** S3 sync. RPO is up to 7 days; should be daily.
- **S3 sync uses `--delete`** — a corrupted/empty local backup folder will wipe S3 too. Consider versioning on the bucket or `aws s3 cp` with date-prefixed keys.
- **No off-site retention policy visible** — only `weekly/` prefix, overwritten each Sunday.
- **`worker.log` is 19 MB and growing** with no logrotate config inside `frappe-bench/logs` (only system logrotate runs, which doesn't touch this path). Will quietly fill disk over months.
- **FastAPI sidecar runs with `--workers` defaulting to 1** — single point of failure for AI chat and Telegram pipeline; one stuck request blocks all chat traffic.
- **`use_redis_auth=False`** — bench redis is on localhost so low risk, but a noauth redis is one nginx misconfig away from being exposed.
- **System redis on :6379 is running but unused by bench** (bench uses :11000/:13000). Dead service consuming RAM.
- **Background worker count = 1** for both short and long workers — scheduled jobs queue up serially. With ~40 doc_events + 8 daily scheduled alerts + Apollo bulk-enrich every 30 min, contention is likely under load.
- **No HTTP-level rate limiting in nginx** that's visible — fail2ban is up but only protects SSH.
- **`unattended-upgrades` is active** on a production box — kernel/package updates can land without notice. Last reboot was Mar 20 (40 days ago) so it's set to security-only, but worth confirming `Unattended-Upgrade::Automatic-Reboot "false"`.
- **Card-scanner service** runs as a separate systemd unit on the same box — extra surface area, shares the 1.9 GB RAM.
- **Cert is a single SAN** (`dgoc.logstop.com` only). If you ever add a second hostname (admin.dgoc.in, api.dgoc.in), certbot will need re-running.
